QA Terminology

Menu


  1. Acceptance testing

    2. Acceptance Testing is a method of software testing where a system is tested for acceptability. The major aim of this test is to evaluate the compliance of the system with the business requirements and assess whether it is acceptable for delivery or not.

  2. Black box testing

    3. Black Box Testing is a software testing method in which the functionalities of software applications are tested without having knowledge of internal code structure, implementation details and internal paths. Black Box Testing mainly focuses on input and output of software applications and it is entirely based on software requirements and specifications.

  3. Bug

    4. A bug is the consequence/outcome of a coding fault.

  4. Clustering

    5. When a small number of modules contains most of the bugs detected or show the most operational failures.

  5. Defect

    6. A defect is an error or a bug, in the application which is created.

  6. Dynamic testing

    7. Dynamic Testing is a software testing method used to test the dynamic behaviour of software code.

  7. Efficiency

    8. Carry out all testing activities in an efficient manner. It is not only about test case execution alone.

  8. Error

    9. Testing is the process of identifying defects, where a defect is any variance between actual and expected results.

  9. Exhaustive testing

    10. Exhaustive testing is a test approach in which all possible data combinations are used for testing. Exploratory testing includes implicit data combinations present in the state of the software/data at the start of testing.

  10. Failure

    11. If under certain circumstances these defects get executed by the tester during the testing then it results into the failure which is known as software failure (example: code have a mistake and result is failure when you want see a database).

  11. Independent testing

    12. Independent testing corresponds to an independent team, who involve in testing activities other than developer to avoid author bias and is often more effective at finding defects and failures.

  12. Integration testing

    13. Integration testing is defined as a type of testing where software modules are integrated logically and tested as a group.

  13. Maintainability

    14. The ease with which a software system or component can be modified to correct faults, improve performance or other attributes, or adapt to a changed environment.

  14. Nonfunctional testing

    15. Nonfunctional testing is defined as a type of Software testing to check non-functional aspects (performance, usability, reliability, etc) of a software application.

  15. Pesticide paradox


  16. Portability

    17. Testing with ease of moving one product or software from one environment to another.

  17. Regression testing

    18. Regression testing is defined as a type of software testing to confirm that a recent program or code change has not adversely affected existing features.

  18. Static testing

    19. Static Testing is a software testing technique which is used to check defects in software application without executing the code. Static testing is done to avoid errors at an early stage of development as it is easier to identify the errors and solve the errors

  19. System testing

    20. System Testing is a level of testing that validates the complete and fully integrated software product.

  20. Test activities

    21. Planning and Control. Analysis and Design. Implementation and Execution. Evaluating exit criteria and Reporting.

  21. Test basis

    22. Test basis or Test Analysis in software testing is a process of checking and analysing the test artifacts in order to base the test conditions or test cases.

  22. Test levels

    23. Include: Unit, Integretion, System and Acceptance in one (on the different levels).

  23. Test objectives

    24. Test Objective is the overall goal and achievement of the test execution, ensure that the software under test is bug free before release.

  24. Testing

    25. Testing is process that ensures that an organization delivers the best products or services possible.

  25. Testing result

    26. Terting result means the results Contractor or the State expects to be achieved by processing the Test Data using the Software.

  26. Unit testing

    27. Unit Tests are conducted by developers and test the unit of code( aka module, component) he or she developed. It helps to reduce the cost of bug fixes since the bugs are identified during the early phases of the development lifecycle.

  27. Usability

    28. Usability refers to evaluating a product or service by testing it with representative users.

  28. White box testing

    29. White box testing is software testing technique in which internal structure, design and coding of software are tested to verify flow of input-output and to improve design, usability and security.
